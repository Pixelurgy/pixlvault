Likeness parameters plan (Feb 11, 2026)

Goal
- Replace N^2 likeness with parameter-driven candidate generation.
- Likeness is only for stacking near-identical images.
- Compute a parameter vector per picture; use it to retrieve candidates via bins/ANN, then verify with embeddings/phash.

Current implementation summary
- LikenessParameter enum in picture.py:
  - SIZE_BIN (index 0)
  - BRIGHTNESS (index 1)
  - Sentinel value: -1.0
- Picture fields:
  - size_bin_index (indexed int)
  - likeness_parameters (float32 vector, LargeBinary)
- LikenessParameterWorker:
  - Computes SIZE_BIN first (exact width/height bin), then BRIGHTNESS.
  - Batches by size bin (same width/height).
  - Uses quality.brightness for brightness.

Parameter ideas to add (commented in enum)
- DATE (created/imported time buckets)
- CONTRAST
- EDGE_DENSITY
- COLORFULNESS / SATURATION
- DOMINANT_HUE
- ENTROPY / TEXTURE
- BLUR_SCORE (variance of Laplacian)
- NOISE_LEVEL

Indexing and candidate retrieval ideas
- Use size_bin_index as a separate indexed field for fast grouping.
- Parameter vector: fixed length float32, sentinel for missing values.
- Build candidate sets by union of “close on at least one dimension”.
- Consider ANN (FAISS preferred over Annoy) on normalized parameter vector.
- For phash: either separate filter or include as 64-dim bit vector appended to params.

Failure handling
- Never allow NaN in parameters; use sentinel and a validity mask.
- If a parameter can’t be computed, leave sentinel and skip that dimension in candidate generation.

Potential next steps
1) Add additional parameters (contrast, edge_density) to enum.
2) Extend LikenessParameterWorker to compute them, still size-bin batched.
3) Add a candidate generator that uses parameter bins and/or ANN (FAISS).
4) Keep final verification with embedding similarity + phash threshold.

Notes
- Size bin should remain a separate indexed field for DB filtering.
- Avoid environment-variable tuning; keep constants in code.
